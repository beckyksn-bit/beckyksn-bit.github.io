sprintf的作用
sprintf是 C 语言标准库（<stdio.h>）中的格式化字符串函数
核心功能是把变量（int/float/char 等），按指定格式转换成字符串，并写入到字符数组（缓冲区）中。也叫做拼接变量。
例如： sprintf(buf, “ADC值：%d，电压：%.2fV”, adc_val, volt);
// 此时buf中的内容是：”ADC值：2048，电压：1.65V”

UART_TypeDef和UART_HandleType的区别
UART_TypeDef：UART 外设的寄存器映射结构体
UART_TypeDef 是 STM32 标准库 / HAL 库中定义的寄存器地址映射结构体，每一个成员对应 UART 外设的一个物理寄存器（如 SR、DR、BRR 等），其首地址直接对应单片机内存中 UART 外设的硬件寄存器地址。
例如：
typedef struct {
__IO uint32_t SR; // 状态寄存器
__IO uint32_t DR; // 数据寄存器
__IO uint32_t BRR; // 波特率寄存器
__IO uint32_t CR1; // 控制寄存器1
// … 其他寄存器（CR2、CR3、GTPR等）
} UART_TypeDef;
// 预定义：将结构体映射到实际硬件寄存器地址（如USART1基地址0x40013800）
#define USART1 ((UART_TypeDef *)0x40013800U)
#define USART2 ((UART_TypeDef *)0x40004400U)//USART1、USART2是指向某特定串口硬件的地址

UART_HandleTypeDef：HAL 库的 UART 操作句柄
UART_HandleTypeDef包含了指向 UART_TypeDef 的指针（Instance），还封装了 UART 的配置参数、运行状态、回调函数等。
标准定义（HAL 库 stm32g4xx_hal_uart.h 中）：
typedef struct __UART_HandleTypeDef {
UART_TypeDef* Instance; // 指向UART寄存器结构体的指针（核心关联）
UART_InitTypeDef Init; // UART配置参数（波特率、数据位、停止位等）
uint8_t* pTxBuffPtr; // 发送缓冲区指针
uint16_t TxXferSize; // 发送数据长度
…
} UART_HandleTypeDef;
其中huart1.Instance=USART1;

uwTick介绍
uwTick=HAL_IncTick(),即从系统启动到现在的时间（ms）；
基本单位：默认 1 毫秒（ms）= 1000 微秒（μs）
数据类型：uint32_t（32 位无符号整数），最大值为 0xFFFFFFFF（约 49.7 天），溢出后会从 0 重新计数

int变量和char变量的本质区别：
当变量值用于字符类输入输出传输场景时，系统硬件会默认将其解析为 ASCII 码；此时不论是char还是int类型都存储与传输的都是ASCLL码值。

pRxBuffPtr是UART接收数据的缓冲区指针，HAL库在接收中断中，会把接收到的每一个字节依次存入该指针指向的内存地址，每存一个字节，该指针会自动向后偏移1位。

串口调试助手模式组合对照表
接收：ASCII 接收数字解析显示为ASCLL码对应字符 eg:发送0x41 串口助手接收为’A’;
发送：ASCII 发送字符解析并发送为ASCLL码值 eg:串口助手发送’A’ 电脑接收到0x41
接收：Hex接收数字解析显示为ASCLL码值 eg:发送’A’ 串口助手接收为0x41;
发送：Hex发送数字解析并发送为对应ASCLL码对应的字符eg:串口助手发送41 电脑接收到’A’;

编程语言中的命名规则

一、驼峰明明法
小驼峰命名法
格式：第一个单词首字母小写，后续单词首字母大写，无下划线。
eg:lowerCamelCase
适用范围：适用场景：绝大多数编程语言中的变量、函数 / 方法、对象属性

大驼峰命名法
格式：每个单词的首字母都大写，无下划线，也叫帕斯卡命名法。
eg:UpperCamelCase / PascalCase
适用范围：适用场景：类、接口、结构体、枚举、全局常量（部分语言），是面向对象编程中的核心命名规范。

二、 常量命名法
这是下划线命名法的变种，也叫全大写下划线命名法，是常量的专属命名规范。
格式：所有字母大写 + 单词之间用 _ 连接。
eg:const MAX_RETRY_COUNT = 3;
适用范围：适用场景：所有编程语言中的常量（包括全局常量、枚举值、配置项中的固定值）。

三、 下划线命名法
下划线命名法的特点是所有字母小写，单词之间用下划线 _ 分隔，风格简洁直观，不易出错。
eg:columns: user_id, user_name, create_time，system_init;
适用范围：函数名，复合变量名。

四、 连字符命名法
连字符命名法的特点是所有字母小写，单词之间用连字符 - 分隔，也叫短横线命名法。
格式：全小写 + 单词之间用 - 连接。
eg:user-info-title {
font-size: 18px;
}

任务调度器

#include “sheduler.h”

void led_proc(void)
{

}//task功能实现

typedef struct
{
void (*task_func)(void);
uint32_t rate_ms;
uint32_t last_run;
}task_t;//定义task结构体类型，包含task功能/周期/last_run时间

static task_t sheduler_task[]=
{
{led_proc,1,0}

};//定义task结构体数组

uint8_t task_num;

void sheduler_init(void)
{
task_num=sizeof(sheduler_task)/sizeof(task_t);
}

void sheduler_run(void)
{
for(uint8_t i=0;i<task_num;i++)
{
uint32_t now_time=HAL_GetTick();
if(now_time>=sheduler_task[i].rate_ms+sheduler_task[i].last_run)
{
sheduler_task[i].last_run=now_time;
sheduler_task[i].task_func();
}
}
}

单片机寄存器

锁存器（latch）—对脉冲电平敏感，在时钟脉冲的电平作用下改变状态
触发器(flip-flop)—对脉冲边沿敏感，其状态只在时钟脉冲的上升沿或下降沿的瞬间改变
寄存器（register）：用来存放数据的一些小型存储区域，用来暂时存放参与运算的数据和运算结果

16位寄存器
1111 1111 1111 1111
GPIOC->ODR&=0x00FF：清除GPIOC数据寄存器高8位，保留低八位
GPIOC->ODR|=0xFF<<8;GPIOC高8位的数据寄存器设为高电平

BSRR：
STM32的BSRR是32位寄存器，按功能分为两个16位区域（对应GPIO的16个引脚Px0 ~ Px15），仅支持写操作，读操作返回无效值。
BRR：
就是BSRR的高16位置零寄存器

第 0 ~ 15位 |BS（Bit Set） |置 1 区域：写 1 对应引脚输出高电平，写 0 无效果| 第8位写1→PC8 置高 |

第 16 ~ 31位|BR（Bit Reset）|置 0 区域：写 1 对应引脚输出低电平，写 0 无效果| 第24位写1→PC8 置低|

eg:GPIOC->BSRR=0x01<<8; //PC8置高电平
GPIOC->BSRR=0x01<<16+8; //PC8置低电平 <==>GPIOC->BRR=0x01<<8;

GPIOD->BSRR = 0x11：仅将 PD0、PD4 置为高电平，PD1PD3、PD5PD15 保持原有电平不变；
GPIOD->ODR = 0x11：强制将 PD0、PD4 置为高电平，同时将 PD1PD3、PD5PD15 全部置为低电平（覆盖所有引脚状态）

关键规则：
无论是BS区域还是BR区域，只有写1有效，写0完全不影响对应引脚。
若同时对某一引脚的BS位和BR位写 1，BS位（置 1）优先级更高。

CubeMX结合keil使用的基础配置

CubeMX结合Keil使用的基础配置
时钟配置：
input frequency–晶振的频率24Mhz(蓝桥杯嵌入式板子) HSE-PLLCLK-APB1、APB2根据题目设置
SYS配置：
蓝桥杯的不用设置，其余情况根据下载器的协议选择
NVIC配置：
System tick timer 优先级设置为0
工程的配置：
固件包选择GW1.4.0版本
keil的配置：
菱形取消勾选；target/微库√；C/C++/优化等级设0； Debug/setting/trace-flash download

.mxproject文件包含项目的元数据和配置
core目录包含核心代码：主程序、系统初始化（时钟、外设）、中断服务例程
Driver目录包含设备驱动程序，通常包含外设的初始化和控制代码（GPIO.UART.I2C等外设的控制接口），设备抽象层（HAL/LL库）
MDK-ARM目录用于存放ARM MDK集成开发环境的相关环境
user存放用户自定义的代码目录
.ioc文件是一个I/O配置文件，通常用于描述控制器的引脚和外设配置
